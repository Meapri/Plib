# Clean-Room Protocol

## Purpose

This protocol lets the project learn from public behavior, open-source software, device evidence, and black-box compatibility probes while keeping the open ALR runtime implementation independent from proprietary or closed runtime internals.

The project may use open-source components where their licenses are acceptable. The project may also run closed or proprietary binaries as optional external benchmarks. The project must not copy closed implementation details into ALR source code.

## Scope

This protocol applies to:

- ALR execution backend implementation.
- Rootfs path virtualization.
- `execve` and child-process continuity.
- Fakeroot behavior.
- Procfs virtualization.
- Graphics and GUI guest shims.
- Any implementation that intentionally overlaps behavior exposed by proprietary or closed runtimes.

## Roles

### Spec Author

The spec author may inspect:

- Public documentation.
- Public source code with compatible or reviewable licenses.
- Project-owned source and tests.
- Runtime command-line behavior.
- Logs, stdout, stderr, exit codes, timing, and device reports.
- ELF metadata that is not used to clone algorithms, such as file type, exported symbol names, dependency names, and documented public ABI information.

The spec author produces:

- Behavior requirements.
- Test cases.
- Compatibility matrices.
- Performance benchmark definitions.
- Failure taxonomies.

The spec author must not provide:

- Proprietary disassembly-derived control flow.
- Disassembly-derived algorithm source.
- Decompiled code.
- Function-by-function algorithm reconstruction from closed binaries.
- Binary patch offsets or instruction sequences copied from a proprietary runtime.

### Implementation Author

The implementation author may inspect:

- Project-owned specifications.
- Project-owned tests.
- Public Linux, Android, ELF, glibc, POSIX, and graphics API documentation.
- Open-source dependencies whose licenses are acceptable for the target artifact.

The implementation author must not inspect:

- Closed-runtime disassembly.
- Decompiled closed-runtime source.
- Private notes that describe closed-runtime algorithms beyond observable behavior.

### Reviewer

The reviewer checks:

- Implementation matches project specs and tests.
- External sources are cited when useful.
- License obligations are recorded.
- Proprietary behavior has not been copied in code or comments.
- Tests describe behavior rather than closed implementation mechanics.

## Source Categories

### Category A: Safe Primary Sources

Use freely for implementation:

- Linux man pages.
- Android NDK documentation.
- AOSP public source where license-compatible.
- glibc public source and manuals where license constraints are understood.
- ELF psABI documentation.
- Khronos EGL, OpenGL ES, and Vulkan specifications.
- POSIX documentation.
- Project-owned rootfs tests and device reports.

### Category B: Open-Source Runtime References

May be studied and reused according to license:

- PRoot and Termux PRoot.
- proot-rs.
- fakechroot.
- libredirect-like projects.
- Mesa, virglrenderer, ANGLE, Venus, and related graphics projects.
- libseccomp examples.

Rules:

- Record license before vendoring or copying.
- Prefer learning interface shape and test ideas over copying code.
- Keep GPL/LGPL components isolated where distribution constraints matter.

### Category C: Closed or Proprietary Binary Probes

May be used only for black-box behavior and optional runtime comparison:

- Command line invocation.
- Public README and release notes.
- Exit status.
- Stdout/stderr.
- Environment variable effects.
- File outputs.
- Timing.
- Device logs generated by running the binary.

Do not use:

- Decompiled source.
- Control-flow reconstruction.
- Instruction-level implementation details.
- Private symbol names as implementation API names unless they are also public product terms.
- Binary patch recipes from the closed runtime.

### Category D: Tainted Reverse-Engineering Notes

If deeper analysis is ever performed for compatibility understanding, it must be isolated from implementation:

- Store under a clearly marked tainted-analysis area outside implementation docs.
- Do not paste into implementation tasks.
- Convert only to black-box behavior tests after review.
- Implementation authors should not read this material.

Current project default: avoid Category D unless explicitly approved.

## Allowed Black-Box Observations

Examples of allowed observations:

```text
Given rootfs R and command /bin/hello:
  backend exits 0
  stdout contains "hello from static arm64 rootfs"

Given command /usr/bin/id with fakeroot enabled:
  stdout contains uid=0 and gid=0

Given path /../../etc/passwd:
  normalized guest path is /etc/passwd

Given guest GUI client sends frames 1..4:
  host receives 4 frames and replies ACK
```

Examples of disallowed implementation leakage:

```text
Closed runtime scans memory using this exact byte pattern.
Closed runtime rewrites instruction X into instruction Y at offset Z.
Closed runtime's function F has this branch structure.
Closed runtime's private metadata format should be copied byte-for-byte.
```

## Specification Rules

Specs should describe:

- Inputs.
- Outputs.
- Environment variables.
- File descriptors.
- Path mapping rules.
- Error behavior.
- Required report lines.
- Compatibility expectations.
- Performance measurements.

Specs should avoid:

- Reusing closed-runtime internal component names when a neutral ALR term exists.
- Describing binary patch internals before the project independently proves wrapper coverage is insufficient.
- Treating any proprietary runtime as an authority for Linux correctness.

## Test Rules

Behavior tests are the clean-room bridge between observation and implementation.

Tests should:

- Use small rootfs fixtures.
- Assert Linux-visible behavior.
- Assert Android safety constraints.
- Compare ALR, PRoot, and optional probe backends through identical commands.
- Capture stdout, stderr, exit code, elapsed time, and relevant file changes.
- Record expected failure classes where parity is not yet required.

Tests should not:

- Assert private implementation names from closed binaries.
- Assert exact log strings from a proprietary runtime unless the log string is part of public CLI behavior.
- Encode proprietary metadata formats.

## Optional Backend Policy

Optional external backends may be included for A/B testing only when:

- They are unmodified.
- Source URL, version, sha256, and license are recorded.
- The app clearly marks them as external.
- Failure does not block the open ALR backend path.
- The same tests can run against PRoot and ALR.

Optional external backends must not become required for:

- Rootfs extraction.
- Host graphics rendering.
- ALR source tests.
- Clean-room implementation tasks.

## Documentation Workflow

For each runtime feature:

1. Write an ALR behavior spec.
2. Add host tests for pure logic.
3. Add Android report-line tests for launch/report plumbing.
4. Add device smoke tests.
5. Compare PRoot baseline.
6. Optionally compare external probe backend.
7. Implement ALR behavior from the spec.
8. Record deviations and known limitations.

## Review Checklist

Before merging ALR runtime implementation:

- The feature has a spec section.
- Tests were added before or with code.
- Public/open-source sources are license-compatible.
- Closed/proprietary binaries were not used as code templates.
- Report output is deterministic enough for device evidence.
- PRoot fallback remains available unless intentionally removed by a later milestone.
- Android safety constraints are preserved.

## Incident Handling

If a change is suspected to include tainted proprietary implementation detail:

1. Stop implementation work in that area.
2. Identify the file, commit, and source of contamination.
3. Revert or rewrite the affected code from clean specs.
4. Add a short note to the relevant design doc explaining the replacement approach.
5. Re-run tests and record the clean source basis.

## Standing Rule

The project goal is not to clone a closed runtime. The goal is to build an open Android Linux runtime whose behavior is validated by public APIs, project tests, and device evidence.
